# 网络重连机制使用指南

## 介绍

在游戏过程中，玩家可能会因网络波动或意外情况导致连接中断。本机制旨在确保玩家能够在连接恢复后快速重回游戏，尽量保持游戏数据的连续性和游戏体验的完整性。


### 常见断线场景

在多人游戏中，玩家会因为各种原因出现网络问题：
- 设备在WLAN与移动数据之间切换
- 设备处于网络信号较弱的区域，导致网络连接不稳定
- 玩家主动切换至后台或结束进程
- 游戏崩溃后重新进入

---

### 重连处理不当的常见问题

网络重连是保障玩家体验连续性的关键技术环节，如果不正确处理，会导致以下异常，最终影响玩家体验：

- 界面显示异常：玩家重连后，血量、能量、金币、积分等关键状态信息显示异常
- 本地状态丢失：客户端通过 RPC 获取的本地状态会丢失（背包物品、技能状态等）
- 功能异常：RPC 调用失败导致交互功能失效
- 同步问题：玩家角色位置、动作等状态不同步

<br>

## 网络状态分类

根据客户端进程是否仍然存在，系统将网络问题分为三种情况：

|状态|客户端状态|关键特征|断网时长|玩家表现|
|-|-|-|-|-|
|网络波动|进程存在|引擎底层自动处理，无需手动处理|0-5秒|游戏画面短暂卡顿或操作延迟|
|静默断线/重连|进程存在|客户端进程仍在运行，网络连接中断，PlayerController保留|5-300秒|游戏内出现“竞赛服务器响应超时，是否尝试重连”提示|
|进程终止断线/重连|进程不存在|应用进程被完全终止（如强制关闭、崩溃），PlayerController需重建|300秒以内|重新登录游戏时，弹出"是否重新进入竞赛"对话框|

---

### 静默重连

静默重连是指客户端与服务器之间的网络连接临时中断，但客户端应用进程并未退出。以下原因都可能导致静默断线：
- 网络信号波动或切换（如Wi-Fi与移动数据切换）
- 进入信号盲区（如电梯、地下室）
- 服务器临时无响应

#### 系统判定规则

- 判定断线：5秒内未收到网络包时，判定为静默断线
- 重连成功：在300秒内恢复网络即可自动重连
- 超时处理：超过300秒仍未重连成功，服务器将玩家踢回大厅

---

#### 玩家表现

出现“竞赛服务器响应超时，是否尝试重连”提示，确认后可尝试进行重连，重连成功后提示消失，游戏继续进行。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/AQ9D6image.png)

---

#### 关键特性

- 保持进程，不存在终止进程
- ConnectionState不变
- Channel不销毁
- 客户端与服务端的游戏对象实例（如 PlayerController、Character等）均被保留，不会销毁重建
- 重连成功后，客户端会收到断线期间DS的最新数据（断线期间未收到的Reliable RPC与值复制的最新值）

---

#### 开发者处理建议

静默重连的底层已处理了大部分网络与数据同步，开发者通常无需处理数据逻辑相关的内容，主要工作集中在客户端重连成功后的表现层恢复。建议操作如下：
- 在客户端重连成功事件中，刷新界面显示（如血量、金币）
- 将本地因断线可能遗留的临时UI（如弹窗、提示）重置为默认状态

---

### 进程终止后重连

进程终止后重连是指客户端应用进程被完全关闭后，玩家重新启动游戏并成功返回原对局的流程。此流程需要重建客户端完整状态。常见场景如下：
- 玩家主动结束游戏进程
- 系统因资源不足清理后台（杀后台）
- 游戏应用发生崩溃（Crash）

#### 系统判定规则

- 判定断线：客户端进程终止
- 判定可重连：玩家若在300秒内重新登录，将会弹出对话框提示"是否重新进入竞赛"
- 重连成功：在"是否重新进入竞赛"对话框点击确定后回到原对局
- 判定超时：超过300秒仍未重连成功，玩家将无法返回原对局

---

#### 玩家表现

1. 游戏进程完全关闭
2. 重新启动并登录游戏
3. 弹出对话框提示"是否重新进入竞赛"，点击确定后返回原对局

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/VqsOqimage.png)

---

#### 关键特性

- 进程终止，重连时创建新的进程
- 重新进行客户端登录DS流程
- 服务端的 PlayerController和 Character对象仍然保留
- 全新的网络连接（NetConnection）
- 客户端所有对象均为新建，因为客户端重启了
- 客户端通过值复制获得的本地状态会保留，但是断线期间值的变化不会同步
- 客户端通过RPC获得的本地状态不会保留

---

#### 开发者处理建议

进程终止后重连需要开发者主动处理全面的状态重建，包括重新请求数据、重建界面等。

服务端示例如下：

``` lua
-- 服务端进程终止后重连回调
function PlayerController:OnDSPlayerControllerReconnected()
    ugcprint("玩家进程终止后重连，完整初始化...")
    
    -- 重新初始化玩家状态
    self:ReInitializePlayerState()
    
    -- 重新发送必要数据给客户端
    self:SendInitialDataToClient()
end
```

客户端示例如下：

``` lua
function PlayerController:OnDSPlayerControllerReconnected()
    -- 重建所有UI
    self:RebuildAllUI()
    
    -- 重新请求所有数据
    self:Server_RequestFullData()
end
```

## 底层事件触发机制

### 相关事件

|事件名称|触发场景|触发范围|说明|
|-|-|-|-|
|PlayerControllerLostDelegate|静默断线和进程终止断线时触发|服务端|玩家断线回调|
|PlayerControllerRecoveredDelegate|静默重连成功时触发|服务端|进程仍然存在，恢复连接|
|PlayerControllerReconnectedDelegate|• 进程终止后重连成功时触发（服务端）<br>• 进程终止后重连时，控制器初始化完成时触发（客户端）|服务端&客户端|• 进程重建，完整重连（服务端）<br>• 客户端重连完成（客户端）|
|ClientPlayerControllerRecoveredDelegate|静默重连成功时触发|客户端|客户端恢复连接|

事件注册的示例代码如下：
``` lua
function PlayerController:ReceiveBeginPlay()
	self.SuperClass.ReceiveBeginPlay(self)
	
	if self:HasAuthority() then
    self.PlayerControllerLostDelegate:Add(self.OnDSPlayerControllerLost, self)
		self.PlayerControllerRecoveredDelegate:Add(self.OnDSPlayerControllerRecovered, self)
		self.PlayerControllerReconnectedDelegate:Add(self.OnDSPlayerControllerReconnected, self)
	else
		self.PlayerControllerReconnectedDelegate:Add(self.OnClientPlayerControllerReconnected, self)
		self.ClientPlayerControllerRecoveredDelegate:Add(self.OnClientPlayerControllerRecovered, self)
	end
end
```

---

### 事件触发时序

静默重连流程：
1. 网络中断超过5秒 → 服务端触发 ``PlayerControllerLostDelegate``
2. 网络在300秒内恢复 → 底层比对玩家状态，发现进程仍存在
3. 服务端触发 ``PlayerControllerRecoveredDelegate``
4. 客户端触发 ``ClientPlayerControllerRecoveredDelegate``

进程终止重连流程：
1. 进程终止 → 服务端触发 ``PlayerControllerLostDelegate``
2. 玩家在300秒内重新登录并确认"返回玩法"
3. 底层比对玩家状态，发现是旧进程（客户端已终止）
4. 进入玩法重连成功，服务端与客户端触发 ``PlayerControllerReconnectedDelegate``

---

### 注意事项

- 网络波动和进程终止时，都会在服务端触发断线回调 ``PlayerControllerLostDelegate``
- 重连时，底层会根据客户端进程是否仍存在，在服务端触发不同的成功回调：
	- 进程仍存在 → 触发 ``PlayerControllerRecoveredDelegate``
	- 进程已终止 → 触发 ``PlayerControllerReconnectedDelegate``
- 在客户端中：
	- 静默重连完成后会触发 ``ClientPlayerControllerRecoveredDelegate``
	- 进程终止的重连，只会在玩家进入游戏，控制器初始化之后触发 ``PlayerControllerReconnectedDelegate``

##  数据同步机制

### 值复制

|数据类型|静默重连|进程终止重连|
|-|-|-|
|DS端Replicated变量|自动恢复最终值|自动恢复最终值|
|本地通过Replication获得的变量|自动恢复最终值|自动恢复最终值|
|玩家基础信息（等级、经验、英雄ID）|自动保留|自动恢复|

---

### RPC调用

|RPC 类型|静默重连|进程终止重连|
|-|-|-|
|Reliable RPC（可靠RPC）|可以接收断线期间发送的RPC|会丢失断线期间的RPC|
|UnReliable RPC（不可靠RPC）|会丢失断线期间的RPC|会丢失断线期间的RPC|

---

### 机制对比与选型建议

|数据同步机制|静默重连|进程终止后重连|适用场景|
|-|-|-|-|
|值复制|自动恢复最终值|自动恢复最终值|状态同步：血量、坐标、金币数量、装备属性等需要保持最终一致性的连续状态|
|RPC调用|• 仅可靠RPC可恢复<br>• 不可靠RPC会丢失|均会丢失|事件通知：背包物品更新、技能状态变化等离散事件，需注意其丢失风险|

---

### 实战建议与示例

在设计中，应区分“状态”与“事件”，并为重连场景做好数据恢复。

**正确做法1**：使用值复制同步状态

``` lua
-- 使用值复制
-- 在 PlayerController 中定义 Replicated 变量
PlayerController.KillReward = 100  -- 会自动同步到重连的客户端
```

**正确做法2**：使用可靠RPC + 客户端缓存（应对重要事件）

``` lua
-- 使用可靠 RPC + 客户端缓存
function PlayerController:Client_OnKillReward(Reward)
    self.ClientCachedKillReward = Reward  -- 本地缓存
    self:UpdateKillRewardUI()
end

-- 在重连时恢复缓存数据
function PlayerController:OnClientPlayerControllerRecovered()
    if self.ClientCachedKillReward then
        self:UpdateKillRewardUI()
    end
end
```

<span style="color:red;font-weight:bold">**错误示范**：使用RPC传递关键状态</span>

``` lua
-- 错误做法：用 RPC 传递重要状态
function MyGameMode:OnPlayerKilled(PlayerController, KillerController)
    -- 这样做的数据会丢失！
    PlayerController.Client_OnKillReward(100)
end
```

## 测试与调试指南

### 测试静默重连

静默重连的测试核心是模拟网络中断但进程存活的场景，并验证状态恢复的完整性。

1. 触发静默断线
	• 物理断网：关闭WiFi/移动网络
	• 代理限流：使用系统工具限制网络（如 Charles throttle）
2. 验证重连流程
	• 保持网络断开状态5-10秒后恢复网络
	• 出现“竞赛服务器响应超时，是否尝试重连”等相关提示
	• 重连成功后UI恢复正常
	• 确认游戏状态（角色血量、金币等）与断线前完全一致
	
---

### 测试进程终止后重连

此测试的核心是模拟进程完全重启的场景，并验证状态的重建与同步。

1. 在玩法中终止游戏进程：
	• 正常关闭：在移动设备上，从多任务视图（Recent Apps）中划掉游戏应用
	• 强制结束：在PC上，通过任务管理器结束游戏进程
	• 模拟崩溃：使用崩溃触发工具
2. 验证重连流程
	• 在进程终止后的5分钟内，重新启动并登录游戏
	• 弹出"是否重新进入竞赛"对话框
	• 点击确定后回到原对局
	• 确认UI和状态等重新初始化
	
---

### 调试工具

在开发阶段，可以通过监听重连事件并打印详细日志来追踪流程和排查问题，示例代码如下：

``` lua
function UGCPlayerController:ReceiveBeginPlay()
    UGCPlayerController.SuperClass.ReceiveBeginPlay(self)
    
    -- 服务端注册回调
    if self.Role == ENetRole.ROLE_Authority then
        -- 断线回调（包括静默断线和进程终止断线）
        self.PlayerControllerLostDelegate:Add(self.OnDSPlayerControllerLost, self)
        
        -- 静默重连回调
        self.PlayerControllerRecoveredDelegate:Add(self.OnDSPlayerControllerRecovered, self)
        
        -- 进程终止后重连回调
        self.PlayerControllerReconnectedDelegate:Add(self.OnDSPlayerControllerReconnected, self)
    else
        -- 客户端注册回调
        
        -- 静默重连回调
        self.ClientPlayerControllerRecoveredDelegate:Add(self.OnClientPlayerControllerRecovered, self)
        
        -- 进程终止后重连回调
        self.PlayerControllerReconnectedDelegate:Add(self.OnClientPlayerControllerReconnected, self)
    end
end

-- 服务端：打印重连类型
function PlayerController:OnDSPlayerControllerLost(PlayerController)
    ugcprint("玩家断线：PlayerControllerLostDelegate")
end

function PlayerController:OnDSPlayerControllerRecovered(PlayerController)
    ugcprint("静默重连成功：PlayerControllerRecoveredDelegate")
end

function PlayerController:OnDSPlayerControllerReconnected(PlayerController)
    ugcprint("进程终止后重连成功：PlayerControllerReconnectedDelegate")
end

-- 客户端：打印重连类型
function PlayerController:OnClientPlayerControllerRecovered()
    ugcprint("客户端静默重连：ClientPlayerControllerRecoveredDelegate")
    self:RefreshUI()
end

function PlayerController:OnDSPlayerControllerReconnected()
    ugcprint("客户端进程终止后重连：OnDSPlayerControllerReconnected")
    self:RebuildAllUI()
    self:Server_RequestFullData()
end
```

## 常见陷阱

### 陷阱1：用 RPC 传递重要状态

错误示例：

``` lua
-- 用 RPC 传递背包数据
function PlayerController:Client_UpdateBackpack(Items)
    self.BackpackItems = Items
end
```

正确示例：

``` lua
-- 使用值复制
PlayerController.BackpackItems = {}  -- Replicated 变量
-- 或者：使用可靠 RPC + 客户端缓存 + 重连时恢复
```

---

### 陷阱2：静默重连后UI未刷新

错误示例：

``` lua
-- 没有重连回调，UI不会刷新
function PlayerController:Client_UpdateHP(HP)
    self.HPText:SetText(HP)
end
```

正确示例：

``` lua
function PlayerController:Client_UpdateHP(HP)
    self.HPText:SetText(HP)
end

function PlayerController:OnClientPlayerControllerRecovered()
    -- 重连后刷新UI
    self:RefreshHPUI()
end
```

---

### 陷阱3：进程终止重连后未重新请求数据

错误示例：

``` lua
function PlayerController:OnDSPlayerControllerReconnected()
    -- 只重建UI，没有重新请求数据
    self:RebuildAllUI()
end
```

正确示例：

``` lua
function PlayerController:OnDSPlayerControllerReconnected()
    -- 重建UI并重新请求数据
    self:RebuildAllUI()
    self:Server_RequestFullData()
end
```

---

### 陷阱4：UnReliable RPC 不可靠

错误示例：

``` lua
-- 用 UnReliable RPC 发送重要数据
function MyGameMode:OnPlayerKilled(Killer)
    PlayerController.Client_OnKillReward(100)  -- UnReliable
end
```

正确示例：

``` lua
-- 使用 Reliable RPC 或值复制
function MyGameMode:OnPlayerKilled(Killer)
    PlayerController.Client_OnKillReward(100)  -- Reliable
end

-- 或者使用值复制
PlayerController.KillReward = 100  -- Replicated
```

## 最佳实践

### 状态数据用值复制

``` lua
-- 推荐：值复制
PlayerController.HP = 100  -- Replicated
PlayerController.MaxHP = 100  -- Replicated
PlayerController.Gold = 0  -- Replicated

-- 避免：RPC 传递状态
function PlayerController:Client_SetHP(HP)
    self.HP = HP
end
```

### 事件用 Reliable RPC + 缓存

``` lua
-- 推荐：可靠 RPC + 本地缓存 + 重连恢复
function PlayerController:Client_OnItemAdded(ItemID)
    if not self.ClientCachedItems then
        self.ClientCachedItems = {}
    end
    table.insert(self.ClientCachedItems, ItemID)
    self:UpdateBackpackUI()
end

function PlayerController:OnClientPlayerControllerRecovered()
    if self.ClientCachedItems then
        self:UpdateBackpackUI()
    end
end
```

## 游戏复杂度与重连处理

- 简单游戏	
	- 只实现进程终止后重连（OnDSPlayerControllerReconnected）即可
	- 静默重连让引擎自动处理
- 中等游戏
	- 实现静默重连 + 进程终止后重连
	- 静默重连处理UI刷新
	- 进程终止后重连处理完整重建
- 复杂游戏
	- 实现完整的重连机制
	- 区分 Reliable 和 UnReliable RPC
	- 本地缓存重要数据
	- 重连时恢复缓存
	- 值复制 + RPC 结合使用

## 完整代码示例

该部分是以 [多人PVE模板](https://developer.gp.qq.com/wikieditor/#/catalog/20190) 为基础，制作的完整的网络同步方案代码。

服务端代码如下：

``` lua
function PlayerController:OnDSPlayerControllerLost(PlayerController)
    ugcprint("玩家断线：PlayerControllerLostDelegate")
end

function PlayerController:OnDSPlayerControllerRecovered(PlayerController)
    ugcprint("静默重连成功：PlayerControllerRecoveredDelegate")
    
    -- 静默重连，数据已由底层同步，只需处理业务逻辑
    self:SendClientWelcomeMessage()
end

function PlayerController:OnDSPlayerControllerReconnected(PlayerController)
    ugcprint("进程终止后重连成功：PlayerControllerReconnectedDelegate")
    
    -- 进程终止后重连，需要完整初始化
    self:ReInitializePlayerState()
    self:SendInitialDataToClient()
end
```

客户端代码如下：

``` lua
-- 静默重连回调
function PlayerController:OnClientPlayerControllerRecovered()
    ugcprint("客户端静默重连：ClientPlayerControllerRecoveredDelegate")
    
    -- 刷新UI
    self:RefreshAllUI()
    
    -- 关闭断线提示
    if self.NetworkTipWidget then
        self.NetworkTipWidget:RemoveFromParent()
        self.NetworkTipWidget = nil
    end
end

-- 进程终止后重连回调
function PlayerController:OnDSPlayerControllerReconnected()
    ugcprint("客户端进程终止后重连：OnDSPlayerControllerReconnected")
    
    -- 重建UI
    self:RebuildAllUI()
    
    -- 重新请求数据
    self:Server_RequestFullData()
end

-- 数据同步示例
function PlayerController:Client_OnItemAdded(ItemID)
    -- 缓存背包数据
    if not self.ClientCachedBackpack then
        self.ClientCachedBackpack = {}
    end
    table.insert(self.ClientCachedBackpack, ItemID)
    
    -- 更新UI
    self:UpdateBackpackUI()
end

function PlayerController:RefreshAllUI()
    -- 刷新所有UI显示
    if self.HPText then self.HPText:SetText(self.HP) end
    if self.GoldText then self.GoldText:SetText(self.Gold) end
    if self.ClientCachedBackpack then
        self:UpdateBackpackUI()
    end
end
```

## 常见问题

### Q1：静默重连和进程终止后重连有什么区别？

- 静默重连：客户端进程仍存在，只是网络中断了。重连时保持PlayerController，值复制数据自动恢复
- 进程终止后重连：客户端进程被终止。重连时创建新PlayerController，需要重新初始化UI和请求完整数据

---

### Q2：如何判断是静默重连还是进程终止后重连？

- 观察回调事件：
	- 静默重连触发 OnClientPlayerControllerRecovered（客户端）
	- 进程终止后重连触发 OnDSPlayerControllerReconnected（服务端和客户端）
- 检查 PlayerState.isReconnected（进程终止后重连为true）

---

### Q3：静默重连后为什么我的UI没有刷新？

因为没有实现 ``OnClientPlayerControllerRecovered`` 回调，在这个回调中主动刷新UI显示即可。

---

### Q4：进程终止后重连后我的数据丢失了，怎么办？

进程终止后重连会丢失断线期间的RPC数据，解决方法如下：
1. 使用值复制替代RPC传递重要状态
2. 使用可靠RPC + 客户端缓存，重连时恢复缓存
3. 在 ``OnDSPlayerControllerReconnected`` 中重新请求完整数据

---

### Q5：UnReliable RPC 什么时候会丢失？

- 静默重连时会丢失
- 进程终止后重连时会丢失断线期间的UnReliable RPC
- 正常游戏中也可能丢失（网络不佳期间）
