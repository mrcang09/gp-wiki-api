# PIE调试失败原因与解决方案

当 PIE 调试失败时，请在 Output Log 中搜索报错日志关键字，匹配下方对应条目，按建议方案排查修复。若多次尝试仍无法解决，请联系官方技术支持。

---

## 一、入口阶段

### `PIE debug failed: remaining debug count is zero, cannot start debug`

每日调试次数已用完，请等待次日次数自动恢复。

### `PIE debug failed: debug precondition check failed`

调试前置条件检查未通过。请确认已登录账号且工程已成功打开。

### `PIE debug failed: invalid type of asset detected, please check output log`

工程中存在不被支持的资产类型。请打开 Output Log 查看具体资产路径，修复或删除对应资产。

### `PIE debug failed: failed to create debug game thread, UploadPath=...`

创建调试线程失败。请检查磁盘剩余空间是否充足，并确认上传目录具有读写权限。

### `PIE debug failed: SubModeID=... does not exist in project settings`

日志中的 SubModeID 在项目设置中不存在。请检查子模式（SubMode）配置，确保工程配置与代码中激活的 SubModeID 一致。

### `PIE debug failed: copy project to upload directory failed, UploadPath=...`

拷贝项目到上传目录失败。请检查磁盘剩余空间和上传目录读写权限是否正常。

### `PIE debug failed: upload file name validation failed`

上传文件名校验未通过。工程中可能存在包含特殊字符（如中文、空格）或路径过长的文件，请将其改为仅含英文、数字和下划线的短文件名。

### `PIE debug failed: lua file validation failed, debug interrupted`

Lua 文件校验失败，调试已中断。请查看 Output Log 中的 Lua 错误详情，修复对应脚本中的语法或逻辑错误。

### `PIE debug failed: survive folder invalid, SurviveDir=..., default choice is trunk/Survive`

Survive 目录无效。请在 DS 设置中重新选择正确的 Survive 目录路径。

### `PIE debug failed: WinServer folder invalid, ServerWorkingDir=...`

WinServer 目录无效。请在 DS 设置中重新选择正确的 WinServer 目录路径。

### `PIE debug failed: WinServer exe invalid, ServerExePath=...`

WinServer 可执行文件无效。请在 DS 设置中重新选择正确的 WinServer .exe 文件路径。

## 二、权限与版本验证

### `PIE debug failed: face verification not performed or not passed`

未完成人脸验证或验证未通过。请重新启动调试并按提示完成人脸验证。

### `PIE debug failed: account login or template login does not have debug privileges`

当前账号或模板不具有调试权限。请联系官方人员申请调试权限。

### `PIE debug failed: no debug privileges`

当前账号无调试权限。请联系官方人员获取调试权限。

### `PIE debug failed: editor version is too low, please upgrade`

编辑器版本过低。请下载并安装最新版本的 UGC 编辑器。

## 三、压缩与上传

### `PIE debug failed: zip failed, ...missing umap or ugcproj file, or folder is read-only`

压缩失败。请检查工程中是否缺少 `.umap` 地图文件或 `.ugcproj` 工程文件，并确认文件夹未被设为只读属性。

### `Compress failed: 7z process creation failed, A7zPath=...`

7z 压缩进程创建失败。请检查日志中 A7zPath 路径下 7z.exe 是否存在，若不存在请修复或重新安装编辑器。

### `Compress failed: 7z returned error code, ReturnCode=..., ZipName=...`

7z 压缩返回错误码。请检查磁盘剩余空间是否充足，并关闭可能占用目标文件的其他程序后重试。

### `Compress failed: zip integrity check failed, ZipName=...`

压缩包完整性校验失败。请检查磁盘空间并清理临时文件后重试。

### `Compress failed: missing required ugcproj file`

缺少必需的 `.ugcproj` 工程文件。请确认工程根目录下存在该文件，如缺失请从备份恢复或重建工程。

## 四、获取上传凭证（Key/Ticket）

### `... rate too fast, please try again later`

操作频率过快。请等待 30 秒至 1 分钟后重新尝试。

### `Daily upload limit reached`

每日上传次数已达上限。请等待次日次数恢复。

### `Ticket is still valid, no need to re-apply`

Ticket 仍有效，无需重复申请。这是正常提示，直接重试即可。

### `Ticket apply count limit reached`

Ticket 申请次数已达限制。请等待 5-10 分钟后重试。

### `No development privilege`

没有开发权限。请联系官方人员为当前账号开通开发权限。

### `Current account cannot upload project, please contact official support`

当前账号无法进行工程上传。请联系官方人员获取上传权限。

### `System error`

系统错误。请重试调试，若多次失败请联系官方支持。

### `API call failed, other type of error`

接口调用失败。请重试调试，若多次失败请联系官方支持。

### `Invalid resource ID`

无效的资源 ID。请确认当前工程的资源 ID 是否正确，如有疑问请联系官方支持。

### `Upload status mismatch`

上传状态不匹配。请重试调试，若多次失败请联系官方支持。

### `Version mismatch` / `Recommended version: ...`

编辑器版本与服务端不匹配。请按日志提示的推荐版本号更新编辑器。

### `Daily whitelist modification limit for this project reached`

本日此工程的白名单修改次数已达上限，请等待次日恢复。

## 五、上传流程

### `PIE debug failed: upload source failed, UploadFilename=...`

上传资源失败。请检查网络连接和防火墙设置，确认编辑器未被阻止联网。

### `PIE debug failed: get key failed`

获取上传凭证（Key）失败。请重试调试，若多次失败请检查网络或联系官方支持。

### `Init multipart upload failed`

初始化分片上传失败。请检查网络连接后重试。

### `Open read file failed, Filename=...`

打开读取文件失败。请检查压缩文件是否存在，以及是否被其他进程占用，关闭相关程序后重试。

### `... Multipart upload part from file failed`

分片上传失败。请检查网络稳定性，切换到更稳定的网络环境后重试。

### `List multipart failed`

列举分片失败。请检查网络连接后重试。

### `Complete multipart upload from file failed`

完成分片上传失败。请检查网络连接后重试。

## 六、DS 启动与下载

### `PIE debug failed: start download failed`

启动下载失败。请重试调试，若多次失败请检查网络或联系官方支持。

### `DS Failed! StartTime=..., DebugID=...`

DS（Dedicated Server）启动失败。请查看 DS 日志定位具体崩溃原因，通常是工程中的逻辑错误导致。

### `PIE debug failed: start debug request failed`

发起调试请求失败。请重试调试，若多次失败请检查网络或联系官方支持。

---

## 七、调试状态查询

### `QueryDebugPrevilege result: ...`

查询调试权限结果。请根据日志中的具体错误信息判断原因，若无权限请联系官方人员获取。

### `Query debug HTTP request failed`

查询调试状态请求失败。请检查网络连接后重试。

### `Illegal API calls found in Lua, please remove the corresponding API calls from the file:`

Lua 脚本中存在不合法的 API 调用。请查看日志中随附的文件列表，逐一打开对应文件，移除使用了非公开或不被支持的 API 调用。

### `Illegal file types found in project, please remove the following files from the project:`

工程中存在不合法的文件类型（如 .exe、.dll、.bat 等）。请查看日志中随附的文件列表，将这些文件从工程中移除。

## 九、Job 任务

### `Job failed, ErrorCode=..., ErrInfo=...`

任务执行失败。请根据 ErrInfo 中的错误描述修复对应问题后重试。

### `Job failed, ErrorCode=...`

任务执行失败（上传/发布阶段）。请重试调试，若多次失败请联系官方支持。

### `Job extracted upload info failed`

任务提取上传信息失败。请重试调试，若多次失败请联系官方支持。

## 常见问题速查

| 问题类型 | 快速建议 |
|----------|----------|
| **权限问题** | 检查是否完成人脸验证，确认账号有调试/开发/上传权限，必要时联系官方 |
| **网络问题** | 检查网络连接和防火墙，切换更稳定网络后重试 |
| **磁盘/文件问题** | 检查磁盘空间、文件读写权限、文件是否存在或被占用 |
| **工程配置问题** | 检查 SubMode 配置、DS 目录和可执行文件路径 |
| **资产/文件错误** | 检查无效资产类型、非法文件类型、不合法 API 调用 |
| **Lua 脚本错误** | 查看 Output Log 修复 Lua 语法或逻辑错误 |
| **版本问题** | 更新编辑器至最新或推荐版本 |
| **频率/次数限制** | 等待时间间隔或次日恢复 |
| **文件名问题** | 修正文件名中的特殊字符或超长路径 |
| **DS 端问题** | 查看 DS 日志定位崩溃原因 |
| **封禁/限制** | 联系官方申诉 |

---

> 排查 PIE 失败时，建议优先在 Output Log 中搜索 `PIE debug failed` 定位具体报错，再对照本文档查找对应解决方案。如按文档操作后问题仍未解决，请联系官方技术支持。
